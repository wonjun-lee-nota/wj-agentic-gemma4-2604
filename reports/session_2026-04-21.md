## Executive Summary

Gemma-4-26B-A4B-it의 RTX 3090 (24GB) 단일 카드 배포를 위한 4-bit quantization을 시도했으나, **fused MoE expert tensor 형식**으로 인해 현존하는 모든 PTQ 라이브러리가 expert weight를 quantize하지 못하는 것을 확인했습니다. MoE expert가 전체 모델 크기의 95%를 차지하므로, 실질적인 크기 축소가 불가능한 상태입니다.

**결론**: 현재 도구 생태계로는 이 모델의 단일 RTX 3090 배포가 불가능하며, vLLM tensor parallelism (multi-GPU) 또는 quantization 라이브러리의 fused-MoE 지원 추가를 기다려야 합니다.

## 2. Model Architecture Analysis

```
Model:          google/gemma-4-26B-A4B-it
Type:           Sparse MoE (Mixture of Experts)
Total params:   ~26B
Active params:  ~4B per token (A4B = Active 4 Billion)
Layers:         30
Experts/layer:  128
Active experts: top-k = 8 per token
Hidden size:    2816
MoE intermediate: 704
```

### Weight Size Breakdown (BF16)

| Component | Size | % of Total |
|-----------|------|------------|
| MoE experts (`experts.gate_up_proj`, `experts.down_proj`) | **45.7 GB** | **94.6%** |
| Other (embeddings, attention, MLP, norms) | 2.6 GB | 5.4% |
| **Total** | **48.3 GB** | 100% |

### Fused MoE Expert Format

Expert weights are stored as **3D fused tensors**:
- `experts.gate_up_proj`: `[128, 1408, 2816]` (BF16) — per layer
- `experts.down_proj`: `[128, 2816, 704]` (BF16) — per layer

이 3D fused format은 추론 최적화를 위한 것이지만, 현존하는 PTQ 라이브러리들은 모두 2D `nn.Linear` weight만 quantize할 수 있어 근본적인 호환성 문제가 발생합니다.

## 4. Trial Results

### Trial t001_gptq_w4_g128

**Method**: GPTQ via gptqmodel 6.0.3  
**Status**: ❌ Skipped — MoE experts not quantized

**Timeline**:
1. **Attempt 1** — transformers 내장 `GPTQConfig` 사용 → `Gemma4Config` has no `use_cache` 오류 (optimum 호환성 문제)

3. **Attempt 3** — gptqmodel 단일 GPU (turtle loading) → **Quantization 성공** (30 layers 완료, ~22분)

**Quantization 결과**:
- Attention layers: ✅ 정상 quantize (qweight int32 format)
- Non-MoE MLP: ✅ 정상 quantize
- **MoE experts: ❌ BF16 유지** (fused 3D tensor 미지원)
- Quantized model size: **46 GB** (원본 49GB 대비 6% 감소)

**Bench 시도**: OOM — 46GB 모델이 24GB GPU에 적재 불가

**Skip 사유**: gptqmodel 6.0.3은 Gemma 4의 fused MoE expert layers를 quantize하지 못함. Attention만 quantize되어 모델 크기 축소가 실질적으로 없음.

## 5. Root Cause Analysis

### 왜 모든 4-bit 방법이 실패하는가

```
                     Gemma-4-26B-A4B-it Weight Structure
                     ===================================

    ┌─────────────────────────────────────────────────────────┐
    │  MoE Expert Weights (fused 3D tensors)                  │
    │  experts.gate_up_proj: [128, 1408, 2816]  × 30 layers  │  95%
    │  experts.down_proj:    [128, 2816, 704]   × 30 layers  │  of model
    │                                                         │
    │  → NOT quantizable by any current PTQ library           │
    ├─────────────────────────────────────────────────────────┤
    │  Attention + Non-MoE MLP (standard 2D nn.Linear)        │
    │  self_attn.{q,k,v,o}_proj, mlp.{gate,up,down}_proj     │  5%
    │                                                         │  of model
    │  → Successfully quantized by GPTQ, NF4, etc.           │
    └─────────────────────────────────────────────────────────┘
```

**모든 PTQ 라이브러리의 공통 한계**: `nn.Linear`의 2D weight tensor만 quantize 가능. Gemma 4의 fused MoE는 `nn.Parameter` (3D)로 저장되어 `nn.Linear`로 인식되지 않음.

### Quantization 라이브러리별 실패 지점

| Library | 실패 지점 | 비고 |
|---------|-----------|------|
| gptqmodel 6.0.3 | `QuantLinear` 래핑 시 3D tensor skip | Turtle loading은 정상, GPTQ 자체는 2D만 처리 |
| bitsandbytes 0.49.2 | `Linear4bit` 변환 시 3D tensor skip | `Params4bit`가 2D weight만 지원 |
| auto-gptq 0.7.1 | transformers 5.x 비호환 | `Gemma4Config.use_cache` 미존재 |
| llm-compressor 0.10.0 | transformers 5.x 비호환 | `TORCH_INIT_FUNCTIONS` import 실패 |
| autoawq | deprecated + torch 2.11+ 필요 | cpp extensions 빌드 불가 |

## 6. Potential Solutions (미실행)

### A. vLLM Tensor Parallelism (가장 실현 가능)
- vLLM으로 4 GPU tensor parallelism 사용
- `--tensor-parallel-size 4`로 49GB 모델을 4장에 분산
- 단일 GPU 24GB × 4 = 96GB로 BF16도 적재 가능
- **한계**: harness의 `bench.py`/`aime_runner.py` 수정 필요 (`ALLOW_HARNESS_EDIT`)
- 측정은 multi-GPU이므로 pinned.yaml의 single-card 제약에 해당

### B. vLLM FP8 Runtime Quantization
- `quantization="fp8"` 옵션으로 FP8 동적 quantization
- vLLM이 MoE expert 포함 전체 모델을 FP8로 변환
- 예상 크기: ~26GB (FP8) — 단일 GPU에는 아직 초과이므로 2-GPU TP 필요
- **한계**: 8-bit이므로 pinned.yaml의 allowed_methods에 `fp8_marlin` 명시 필요

### C. GGUF 변환 (llama.cpp)
- `llama.cpp`의 `convert-hf-to-gguf.py`로 GGUF 형식 변환
- GGUF는 MoE expert를 개별적으로 quantize하는 것을 지원
- Q4_K_M 등으로 ~13GB 달성 가능
- **한계**: harness가 `transformers` 기반이라 GGUF 모델 로딩 불가

### D. Quantization 라이브러리 업데이트 대기
- gptqmodel, vLLM의 llm-compressor 등이 fused MoE 3D tensor 지원 추가 시 재시도
- Gemma 4는 2026년 출시 모델이라 도구 지원이 아직 성숙하지 않음

## 7. Recommendations

1. **단기 (즉시 실행 가능)**: vLLM + 4-GPU tensor parallelism으로 BF16 추론. Harness를 `ALLOW_HARNESS_EDIT=vllm_tp4_moe_compat`로 수정하여 vLLM 백엔드 사용. AIME 평가는 vLLM의 `SamplingParams`로 동일 프로토콜 재현 가능.

2. **중기**: gptqmodel 또는 llm-compressor의 fused-MoE quantization 지원 릴리스를 추적. gptqmodel의 GitHub에 이슈 등록하여 Gemma 4 fused MoE 지원 요청.

3. **장기**: Google의 Gemma 4 공식 quantized checkpoint 출시를 모니터링. 모델 제작사가 공식 quantized 버전을 제공하면 가장 안정적인 결과를 얻을 수 있음 (단, harness의 R7 규칙에 의해 pre-quantized download는 금지이므로, 그 시점에 정책 검토 필요).
